What This Is

This project treats UMLS as the stable concept identity layer and builds the successor retrieval product around it: evidence-backed CUI/view vectors, Elasticsearch indexes, source provenance, and reviewed search judgments.

It is not another MRCONSO synonym expansion. A release-quality artifact should prove that it retrieves the right CUI for real biomedical language and can show the evidence behind each result.

How to Use It

  1. Open this page in a browser, not just the IDE editor.
  2. Click Open integrity assessment to check files, counts, and scale assumptions.
  3. Click Open search quality assessment to run searches and judge successor-index quality.
  4. In the Artifacts section, choose the files listed below.
  5. Leave Max parsed lines as 0 to scan the whole file.
  6. Click Analyze selected files.
  7. Read Integrity Checks first. Failures are real problems; warnings are things to inspect.

Search Quality Test

The easiest version is server-backed, so you do not load files manually. The current canonical server command is maintained in docs/technical_pipeline.md; it includes all five topic chunks, the PubMed bulk pilot, Elasticsearch ANN search, and the UMLS label fallback index.

If the server is already running, open http://127.0.0.1:8766/. Check http://127.0.0.1:8766/api/status to confirm the active vectors, evidence files, label index, and judgment path.

  1. Use the server-backed page above for normal testing.
  2. Use the static search quality assessment only when you need a no-server fallback.
  3. Use the SapBERT/CLS vector JSONL and the concept document JSONL.
  4. Run one query at a time, inspect the labels and evidence snippets, and mark results as relevant, partial, or wrong.
  5. For a repeatable benchmark, add one query per line in the query set box. If you know the target CUI, add it after a tab or comma.
  6. Use the exported judgment CSV as the start of the release evaluation set.

What Good Looks Like

  • The cumulative Elasticsearch alias qe-scaling-sapbert-cls includes all reviewed chunk indexes.
  • The current five-chunk plus PubMed bulk pilot prototype has 219,367 SapBERT vector records loaded by the assessment server.
  • The source evidence shown in the UI comes from build/search_quality_provenance.sqlite, currently 667,003 display provenance references.
  • Vector dimensions show one value, currently 768.
  • DRG source rows show 0.
  • Parse errors show 0.
  • Queries return plausible CUIs with PubMed or Europe PMC provenance, not just high vector scores.
  • No chunk becomes release-quality until reviewed judgments show acceptable precision.
