How To Read This Page

The search-quality reports are concept-retrieval tests. Each row sends text to the local /api/search endpoint and checks whether the ranked results contain the intended UMLS concept IDs. The page tracks both quality problems and process changes, so the same words repeat across the snapshot, backlog, timeline, and experiment table.

What PubMed Means Here

For the PubMed benchmark, the input is a reviewed PubMed title plus abstract stored in a TSV file. The benchmark sends that long text to the local search API as the query. It is not doing a live PubMed search during the test, and it is not sending full articles to an external service. The API tries to return the medical concepts mentioned in the abstract.

Focused Versus Approved PubMed

The focused PubMed set is a seven-abstract practice slice chosen because it stresses long-document behavior. The approved PubMed suite is the broader 13-abstract reviewed set. "Current" means it has been rerun with the latest code and data; the older 3/13 result is kept as history but should not be treated as today's status.

What A Pass Means

"Top 10" means the concept appears somewhere in the first 10 answers. "Strict top 10" or "rows complete" means every expected concept for that row appears in the first 10. "Top-on-target" means the first answer is at least an intended or accepted concept. "Wrong-first" means the row found useful concepts but put a known bad concept first.

Term Plain Meaning
CUI A UMLS concept identifier, such as C0034065 for Pulmonary Embolism. Tests use CUIs so label wording changes do not hide whether the right concept was found.
Lane A named group of related tests, such as short user searches, patient-portal messages, PubMed abstracts, or DailyMed source checks.
Probe A diagnostic test set. It can reveal or measure a weakness without becoming a release-blocking gate yet.
Gate / smoke A regression check that must stay acceptable after a change. "Smoke" means a smaller check used to catch obvious breakage quickly.
Baseline The comparison run for a test set. "Current baseline" means the latest rerun with current code and data. "Stale baseline" means the number came from an older system state.
Disallowed CUI A known bad concept for a row. If it appears in the first 10 answers, the row exposes a false-positive or literal-drift problem.
Acceptable alternative A reviewed concept that is allowed to satisfy the intended target only in a specific context, such as a dose-form drug concept standing in for the ingredient when the abstract uses that exact wording.
Good / mixed / poor Good means the row found everything expected and avoided known bad first answers. Mixed means some expected concepts or ordering still need work. Poor means the row failed badly enough to drive a fix.
SQI, SQB, WFN SQI is an iteration record, SQB is a backlog task, and WFN is a reviewed wrong-first/noise issue.
Opt-in probe A signal kept available for experiments or an explicit UI mode, but not loaded into default patient-facing results.
Relation-derived A candidate inferred from relationship data rather than directly mentioned in the query text. These can be useful in exploration views but noisy in the main answer list.

Cap Rule

Do not report stronger than 76% while both realistic note formats and approved PubMed long abstracts score below 60. This keeps green short-query, source, consumer, and top-answer lanes from hiding major breadth failures.

Update Rule

Append a row whenever a shipped iteration changes user-visible behavior, a major benchmark, a routine regression check, source transparency, or a top known quality risk. Rerun the scorecard, record the raw weighted score and any active cap, and do not raise the percentage for a narrow recall win if visible noise or wrong first answers get worse.

Process rule added

The loop now requires a test-quality step, not just a code fix. If a run finds a weakness, the iteration must either add a new test, improve an existing test, tighten a threshold, or record why the existing tests already cover that weakness.

What Changes

Future loop entries should include what changed in the test suite, not only what changed in the app. A fix is not complete if the weakness can disappear from visibility in the next iteration.

How It Is Scored

The loop can satisfy this step by adding a focused regression test, promoting a recurring miss into a suite lane, updating a benchmark row, adding an acceptable alternative, or tightening a release threshold.

Exception

If no test change is appropriate, the loop must say why. That exception should be rare and explicit.
