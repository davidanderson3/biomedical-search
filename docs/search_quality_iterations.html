Test Documentation Contract

Every manual or Codex-run search-quality test should leave behind the same four answers: what weakness the test found, what test evidence exposed it, what solution should be used, and whether that solution has shipped. Runtime changes also need the standing clinical smoke result before they are marked shipped; broad ranking/runtime and release-quality changes need the 50-query rotating smoke with gates. Start each iteration by reading the progress log's latest current-loop, prioritized backlog task IDs, open-work, and chronology sections so the next problem comes from the current status. If this is the first iteration started on a new calendar day, first add a lay-language summary of the previous day's shipped improvements to the daily accomplishments page unless that summary already exists. Keep the progress log linked to that page. Iterations that change headline status must also update the readable progress log before they are marked shipped. Every iteration must refresh the progress-log backlog table by task ID or explicitly confirm that it is still accurate. Product-impacting iterations must update the whole-product quality percentage history in the progress log. Evidence-impacting iterations must update the evidence bucket page.

Start From Log Use the latest progress-log cards, backlog task IDs, open work, and chronology to choose the next problem.
Daily Summary On the first iteration of a new day, summarize the previous day's shipped improvements in the daily accomplishments page.
Weakness Found The failure, miss, false positive, audit row, or benchmark gap.
Test Evidence The command, smoke run, API check, audit, or manual review that exposed it.
Recommended Solution The expected behavior or bounded fix route before changing code or data.
Implementation Status Shipped, proposed, blocked, reverted, or follow-up needed with decision notes.
Backlog Refresh Update the backlog table by task ID every iteration, or explicitly confirm that the current priorities still stand.
Product Score Update the whole-product quality percentage when user-visible quality, major gates, source trust, or top risks move.
Evidence Buckets Update the evidence bucket page when evidence is acquired, tested, promoted, rejected, rerouted, or made pending.
